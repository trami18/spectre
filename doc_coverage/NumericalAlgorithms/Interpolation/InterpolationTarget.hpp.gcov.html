     124             : /// ### Interpolation with time-dependent CoordinateMaps
     125             : ///
     126             : /// Each set of points to be interpolated onto is labeled by a
     127             : /// `temporal_id`.  If any step of the interpolation procedure ever
     128             : /// uses a time-dependent `CoordinateMap`, then it needs to grab
     129             : /// `FunctionOfTime`s from the `GlobalCache`.  Before doing so, it
     130             : /// must verify that those `FunctionOfTime`s are up-to-date for the
     131             : /// given `temporal_id`.
     132             : ///
     133             : /// Note that once the `FunctionOfTime` has been verified to be
     134             : /// up-to-date for a particular `temporal_id` at one step in the
     135             : /// interpolation procedure, all subsequent steps of the interpolation
     136             : /// procedure for that same `temporal_id` need not worry about
     137             : /// re-verifying the `FunctionOfTime`.  Therefore, we need only focus
     138             : /// on the first step in the interpolation procedure that needs
     139             : /// `FunctionOfTime`s: computing the points on which to interpolate.
     140             : ///
     141             : /// Each `InterpolationTarget` has a function
     142             : /// `InterpolationTargetTag::compute_target_points` that returns the
     143             : /// points to be interpolated onto, expressed in the frame
     144             : /// `InterpolationTargetTag::compute_target_points::frame`.  Then the
     145             : /// function `block_logical_coordinates` (and eventually
     146             : /// `element_logical_coordinates`) is called to convert those points
     147             : /// to the logical frame to do the interpolation.  If
     148             : /// `InterpolationTargetTag::compute_target_points::frame` is
     149             : /// different from the grid frame, and if the `CoordinateMap` is
     150             : /// time-dependent, then `block_logical_coordinates` grabs
     151             : /// `FunctionOfTime`s from the `GlobalCache`.  So therefore any Action
     152             : /// calling `block_logical_coordinates` must wait until the
     153             : /// `FunctionOfTime`s in the `GlobalCache` are up-to-date for the
     154             : /// `temporal_id` being passed into `block_logical_coordinates`.
     155             : ///
     156             : /// Here we describe the logic used in all the Actions that call
     157             : /// `block_logical_coordinates`.

     159             : /// #### Interpolation using the Interpolator ParallelComponent
